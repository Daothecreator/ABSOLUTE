# ABSOLUTE: архитектура платформы приватности и независимого знания нового поколения

## Исполнительное резюме

Этот документ формализует «полную архитектуру ABSOLUTE» как надстройку над текущим ядром проекта:

- реалистичный bootstrap-поток («без отдельного установщика», но с явным согласием на привилегии ОС);
- прозрачная и проверяемая цепочка поставки (reproducible builds + TUF + Sigstore/Rekor);
- безопасный терминал с white-list командами в песочнице (Wasm/WASI + capability model);
- независимый поиск и энциклопедия аудитов с криптографической неизменяемостью и юридическими ограничениями по PII.

## Неподвижные принципы

1. **Минимизация данных по умолчанию**: собирать только то, что нужно для защиты.
2. **Локальность и прозрачность**: хранение локально по умолчанию, полное журналирование действий компонентов.
3. **Проверяемость**: пользователь и аудиторы могут проверить происхождение артефактов и решений политик.

## Юридические границы для поиска и энциклопедии

Для режима «поиск всегда находит» применяется безопасная интерпретация:

- система никогда не возвращает «пустой экран»,
- но блокирует doxxing-класс запросов и индексацию персональных данных частных лиц,
- в сомнительных случаях предлагает безопасные альтернативы (агрегированные/обезличенные источники, официальные реестры, публичные решения без персональных идентификаторов).

Энциклопедия аудитов хранит **проверяемые первичные артефакты**, а не произвольные утверждения:

- CID/хеш документа,
- provenance-метаданные,
- тип и статус источника,
- ссылку на первичную публикацию,
- подпись и запись прозрачности.

## Развёртывание: честная модель «без установщика»

ABSOLUTE Setup (единый bootstrap-артефакт) выполняет только детерминированные шаги:

1. проверяет подписи и включение в transparency log;
2. проверяет воспроизводимость/верифицируемость сборки;
3. запрашивает режим: **Network-only** или **Kernel-guard**;
4. создаёт и шифрует локальный **ABSOLUTE Vault**;
5. включает панель прозрачности с журналом действий компонентов.

## Платформенная матрица (реалистичный максимум)

- **Linux**: eBPF наблюдение + LSM eBPF enforcement + XDP/tc/cgroup hooks (требуется root).
- **macOS**: Endpoint Security + Network Extension через system extensions (требуется админ-одобрение).
- **Windows**: WFP callout driver + user-mode сервис (обязательная подпись kernel-драйвера).
- **Android**: VpnService как baseline без root; расширенный режим только на привилегированных устройствах.
- **iOS/iPadOS**: NetworkExtension/per-app VPN в пределах разрешённой модели, без стороннего kernel-level мониторинга.

## Ядро политики и доверия

### Унифицированный контракт событий

Минимальный общий формат событий (для Linux/macOS/Windows/Android/iOS):

- `process.exec`, `process.inject`, `fs.read`, `fs.write`, `net.connect`, `net.dns`, `sensor.camera`, `sensor.mic`, `crypto.cert.pin_violation`.

Обязательные поля:

- идентификатор процесса и цепочка предков,
- подпись/идентичность бинарника (если доступно),
- цель доступа,
- объём/частота,
- решение политики (`allow|deny|alert`),
- обоснование и след доказательств.

### Политики и расширения

- trusted core: STLC policy engine (формально верифицируемая база);
- переносимые модули: WebAssembly;
- ограничение возможностей: WASI + UCAN capability grants.

## Vault и криптография

- KDF: **Argon2id**;
- шифрование: **AEAD ChaCha20-Poly1305**;
- восстановление: mnemonic recovery phrase;
- логи: append-only + периодические Merkle roots, подписанные ключом пользователя.

## Supply chain и обновления

Минимальный «тройной контур»:

1. **Reproducible builds** (как цель поставки);
2. **TUF** для защищённых обновлений;
3. **Sigstore/Rekor** для прозрачности событий подписи/публикации.

## Безопасный терминал ABSOLUTE

Каждая команда оформляется как Wasm-капсула с манифестом:

- `describe()` — что делает;
- `required_caps()` — какие capability-права нужны;
- `run(args)` — запуск с лимитами CPU/RAM/IO/timeout.

Команды не получают доступ к сети/файлам вне явно выданных capability.

## Критерии MVP «без заглушек»

1. **Linux**: загружается eBPF, идёт поток событий, работает минимум одно LSM-deny правило с UI-уведомлением.
2. **macOS**: активируется system extension, приходят ES events, отображается решение политики.
3. **Windows**: установлен и применён WFP-компонент, блокировка/логирование работают.
4. **Android**: поднят VPN интерфейс, видны исходящие соединения, работает доменный блок-лист.
5. **Search**: результат всегда непустой (точные/близкие/кандидаты), при этом PII-доступ ограничивается.
6. **Audit Encyclopedia**: документы адресуются по CID/хешу, provenance обязателен, скачивание PDF поддержано.

## Примечание по следующему шагу

Документ задаёт архитектурные требования и интерфейсные контракты. Следующий инженерный шаг — создать конкретные схемы (`absolute-event.schema.json`, `terminal.wit`, `capabilities.json`) и привязать их к существующим компонентам проекта.
